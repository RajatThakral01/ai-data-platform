# 🧠 AI Data Platform: The Definitive Technical Blueprint

This document is the ultimate, comprehensive technical reference for the AI Data Platform. It details the architecture, module workflows, state management, session orchestration, AI systems (RAG, LLM chains), and local execution logic.

---

## 🏗️ 1. Architecture: The Decoupled Full-Stack System

The platform is split into two distinct high-performance layers, orchestrated tightly via Docker or local runtime logic.

### A. Frontend: The Interaction Layer (Next.js 14)
The frontend is built with **Next.js 14** using the **App Router**, designed to seamlessly feed complex data representations and insights directly to the user.
- **Styling**: **Tailwind CSS** for a highly adaptable UI, combined with **Lucide React** for iconography.
- **UI Components**: Built on **ShadcnUI** (which utilizes Radix UI primitives) to ensure accessibility and consistent design.
- **State Management**: A Centralized **React Context API** (`frontend/lib/store.tsx`) manages cross-module states. It rigorously tracks:
  - `sessionId`, `filename`, `columns`, `rowCount`
  - Active module pointers (`currentModule`)
  - Modular results (`edaResults`, `cleaningResults`, `mlResults`, `insightsResults`)
  - A comprehensive query history (`nlHistory`)
- **API Communication**: Unified API definitions inside `frontend/lib/api.ts` utilizing Axios, interceptors, and a dynamic backend routing schema.
- **Data Visualization**: Leverages a mix of **Plotly.js**, **ECharts**, and **Recharts** to render highly dynamic JSON chart configurations generated by the AI agent logic on the backend.

### B. Backend: The Intelligence Layer (FastAPI)
The backend is a **FastAPI** application (`backend/main.py`) engineered for high concurrency, AI parallel processing, and heavy analytical lifts.
- **Concurrency**: Leverages Python's `async/await` methodology for all LLM calls, and `BackgroundTasks` for non-blocking operations like RAG chunk indexing and background report generation.
- **Session Persistence (`backend/session_store.py`)**: Uses a **Session-over-File** model. Instead of repeatedly loading massive CSV/Excel files into memory, the backend retains a file pointer associated with a dynamic `session_id`.
  - **Scaling**: Uses **Redis** for distributed state management to scale across multiple workers, alongside **Supabase** tracking for long-term session logging, with an in-memory python fallback mechanism for local testing.

## 🤖 2. The Core AI Engine (Agentic Logic)

### A. Triple LLM Routing (Failover & Optimization)
Located entirely in `backend/llm/client_factory.py`, this is the platform's execution core. To ensure 100% uptime and cost optimization, it employs a cascading failover architecture:
1. **The Factory**: Automatically wraps calls in a unified interface that supports `max_tokens`, `system_prompt`, and model selection.
2. **The Routing Chain**:
   - **🟢 Tier 1 - Groq (Llama-3.3-70b-versatile / 3.1-8b-instant)**: The primary engine. Blazing fast (generating tokens at <100ms) with extremely high logic-per-second ratios.
   - **🟡 Tier 2 - Gemini (gemini-2.0-flash)**: The primary fallback. It's invoked dynamically when Groq hits a rate limit or a timeout, leveraging Google's 1M+ context window.
   - **🔴 Tier 3 - Ollama (mistral/llama3)**: The ultimate local safety net. Secures offline environments or zero-budget operations.
3. **Execution Safety**: Code execution prompts (like Next.js frontend code or Python data manipulation logic) are aggressively routed to the most capable logic models (70B/Flash), while simple summarizations leverage smaller instant models (8B).

### B. RAG (Retrieval-Augmented Generation) Pipeline
Located in `backend/rag/` (including `vector_store.py` and `document_processor.py`). Allows for conversational querying across massive datasets without blowing out context windows.
- **Storage Dual-Stack**: Uses **Supabase (pgvector)** as the primary highly available vector store, and falls back gracefully to a localized **ChromaDB** container if Supabase is unavailable.
- **Hybrid Retrieval**: Extracts top dataset chunks and metadata simultaneously via RPC matching bounds, pushing relevant row/column context directly into the LLM context.
- **Embedder**: Utilizes the `sentence-transformers` library to seamlessly convert column schemas and row content into multi-dimensional vectors.

## 📁 3. System Modules: Deep Dive

### 1. 📊 Smart EDA (`routers/eda.py`)
- **Automated Profiling**: Parses Pandas DataFrames to detect missing values, uniqueness boundaries, categorical/numerical variance, and cardinality distributions.
- **Visualization Logic**: Uses a native Python helper (`utils/visualizations.py`) to output raw **JSON-serializable chart configs** (Plotly payload format) natively understood by the frontend context without requiring heavy manual mapping.

### 2. 🗣️ NL Query Engine (`routers/nl_query.py`)
A highly resilient **"Text-to-Code" engine**:
- **Phase 1: Context Awareness**: Ingests active metadata directly out of memory based on the `session_id`.
- **Phase 2: Agent Generation**: Sends the natural language question alongside a sanitized structural schema to the designated LLM.
- **Phase 3: Code Execution**: Evaluates the LLM-returned Python manipulation script safely inside the local backend context, catching output tracebacks and recovering automatically.
- **Phase 4: Serialization**: Parses final `result_df` representations into renderable React formats.

### 3. 🤖 ML Recommender (`routers/ml.py`)
- **Goal Engine**: Dynamically senses Target Variable parameters to distinguish whether a user requires Classification or Regression architecture.
- **Feature Engineering**: Advises the Next.js frontend on categorical columns requiring one-hot-encoding, numerical targets needing normalization, etc.
- **Recommendation Algorithms**: Suggests and frames exact configurations for XGBoost, Random Forest, Logistic Regression, etc., derived based accurately on dataset dimensionality.

### 4. 💡 LLM Insights (`routers/insights.py`)
- **Pattern Recognition**: Operates asynchronously to consume raw `describe()` statistical matrices.
- **Narrative Outputs**: Transforms flat correlation matrices and standard deviations into human-readable narratives (e.g., *"Sales dipped 40% uniformly on Thursdays matching extreme variance..."*).

| Storage Boundary | Primary Tech | Target | Purpose |
|------------------|--------------|---------|---------|
| **Raw Datasets** | Local File I/O | `/app/uploads` | Intermittent storage handling multipart CSV/Excel ingestion blobs. |

## 🚀 7. Platform Tooling & Expansion Flow

When writing new features, follow these standardized internal workflows:
- **Backend Expanders**: Always use the global `store_dataset` methodology in `vector_store.py` for text saving, and interface purely with `get_llm_client` or `client_factory.py` for text generation. 
- **Frontend Expanders**: New tabs belong under `app/dashboard/<module_name>/page.tsx`. New context variables belong in `lib/store.tsx` (add to both the `StoreState` interface and the initial `StoreProvider` dictionary hook).
To add a new feature:
1. **Define the Router**: Add a file in `backend/routers/`.
2. **Register**: Add the route to `app.include_router` in `backend/main.py`.
3. **Client**: Use `LLMClientFactory.get_client()` for any AI logic.
4. **UI**: Add a new `.tsx` file in `frontend/app/dashboard/` and a link in `Sidebar.tsx`.
